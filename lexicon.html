The Lexicon

Our coined terms — the methodology, architecture, and coding vocabulary (and the tradition-taxonomy codes). The names we use, defined.

The running list of the project's coined names — architecture, methodology, coding

vocabulary, cohort mechanisms, naming layers. The point is so we **don't forget our own

code-names** across a multi-year, many-session horizon.

How to maintain this doc. When a new name is coined (a new acronym, an architecture

nickname, a methodology term, a naming-layer lock), add a one-line entry here in the right

section + point to where it's defined (CLAUDE.md §, Decision #, or the lock-doc). Keep

each entry tight — this is an index, not the spec. The authoritative definition lives in

the cross-referenced doc; this is the lookup table.

Terms marked [internal-only] are NOT used in public-facing bio / Substack / talks.

Terms marked [HYPOTHESIS] are hypothesis-layer, NOT findings (per CLAUDE.md § 2.1).

Casing convention: garden metaphor-words are lowercase (seed · flower · garden · rain · graft · trellis · germination); imported/proper terms are capitalized (Sitz · Metaxy); acronyms are UPPERCASE (CRADLE · PRISM · FOUNDRY — they expand; rain/graft do NOT). W-numbers (W10/W13) are the workstream labels; the metaphor-word inside is lowercase.

5a. The defense-paced workstreams (W1–W8)

The 8 build workstreams locked S100 evening (.claude/SESSION_TODOS.md top-section). Run order: W8 → W7 → W2 → W1 → W5+W6 → W3+W4. Every STANDUP "today's aim" + STANDDOWN "what we did" is filtered through "did this advance W1–W8?"

The logic of the run order is dependency, gated by defense-criticality — NOT "start-first" (that's the output) and NOT "furthest-along": (1) W8 the correctness gate runs first — nothing downstream is trustworthy if the T-stage coding is wrong, so gate-before-build (W8 is the least-finished, not the most); (2) producer-before-consumer — W2 plants coded seeds before W1 consumes them into cross-seed rhymes; (3) W1 is the defense OBJECT everything converges on — and it runs 4th despite being the most build-ready ("push-button"), which is the clean proof the order is not by progress; (4) W5/W6 (mechanical staging) + W3/W4 (background infra) don't gate the defense object, so they fill around the critical spine. Sanity check for any reorder: "what does this workstream need before it can produce its output?"

> W-IDs are stable, NOT sprint-scoped — don't prefix them (13W2). W1–W8 is one set locked for the defense horizon (~10 months / several sprints); W8 is still W8 next sprint, so a sprint prefix would wrongly imply the set resets every 10 sessions (re-introducing the "is this the same thing?" confusion). The clean "when am I" handle is the session number (e.g. S101), which is monotonic and unambiguous. The sprint counter is not a reliable prefix anyway — it has drifted (retrospective files run sprint_5→6→7→8→9 then lump S81–S100; loose todo mentions span 10–14); a counter-cleanup is a worthwhile small tidy-up later, but the W-IDs should stay bare regardless.

#### Run-naming convention (S101 lock — retires "IRR101 vN" for these runs)

The problem with "IRR101 v4": "IRR" falsely implies a manual 5-LLM round, and "vN" reads like "same code, next iteration" when the thing that actually changed is coverage (60 → 321 passages). The fix: a run name encodes what (action) · which seed · how-many-passages (coverage) · mechanism · date — never a bare version number.

Run kindNaming patternExampleReads as
Thread-assembly (assign T-stages to a seed's passages)THREAD-<SEED>-<N>-<date>THREAD-ROMANS-321-2026-06-01"threaded the Romans seed, 321 passages, today"
T-stage correctness check (score assembler tags vs cohort)TCHECK-<SEED>-<N>-<mech>-<date>TCHECK-ROMANS-newN-SDK-…"checked the Romans tags, N new atoms, SDK cohort"

Rules: (a) <N> is the passage count, not a version — 321 vs 60 instantly shows it's not the same batch (your ask: "show the range 60→321"). For a delta-only run use the range form (60to321). (b) <mech> is from the vocabulary lock (SDK / 3IRR / 3P-SDK / 5IRR) so a run can never be mistaken for a manual IRR. (c) No "vN" — if the code genuinely changed, that's a git commit / PASS2_VERSION in the script, not the run name. The run name is about what was processed, not which code revision. 8. Plain-English computing terms (BORROWED from engineering — NOT project-coined)

Unlike sections 1–7 (our coined names) and the TRADITION_GLOSSARY (traditions' own terms), these are generic computing words Claude uses in passing. Added S102 (2026-06-02) per researcher request — "useful for me to educate myself." Each entry: what it is + how it shows up here.

8a. Version control — git (how we save + share every change)

  • git — the system that records every change to the project's files over time. Infinite undo + a complete history of who changed what, when.
  • commit — one labeled save-point: a snapshot of changes plus a message describing them. ("S102 close — Cave T10 filled…" is a commit message.)
  • branch — a parallel line of work. You can build/experiment on a branch without touching the official version. Ours are named claude/....
  • main — the canonical, official branch — the "real" version of the project. New sessions start from main.
  • push — upload your commits from your computer to the shared server (GitHub).
  • PR (pull request) — a written proposal to merge a branch's changes into main, with a summary. (PR #586 was today's close.)
  • merge — actually combine a branch's changes into another (usually into main).
  • rebase — replay your changes on top of the latest version, so history stays tidy and you're building on current work (the dispatch "rebase-before-push" rule, § 13 Mechanism 10).
  • cherry-pick — grab one specific commit or file from another branch without taking everything else (how I pulled the SB deliverables in).
  • fast-forward — update your local copy to match the server when nothing conflicts (the step that lets your Obsidian see merged work).

8b. Command-line / shell (typing commands straight to the computer)

  • shell / terminal / command-line — the text interface where you type commands instead of clicking. Where all the ./scripts/... runs happen.
  • grep — search text for a pattern. "grep for X" = "find every line containing X." One of the most-used tools; it's how I check whether something already exists before building it.
  • regex (regular expression) — a tiny pattern-language for describing text (e.g. "any word starting with cave"). Our drill configs (the flower pools) are built from regex patterns.
  • pipe ( | ) — feed one command's output straight into the next, like an assembly line. grep X | sort = find X, then sort the results.
  • background process / nohup / & — run a long job that keeps going after you walk away (the overnight flower-coding is one).
  • PID (process ID) — the number the system assigns a running program so you can check on it or stop it (the overnight coding is PID 63806).
  • stdout / stderr / log — where a program prints its normal output / its errors / a saved file recording both. When I "check the log," that's the saved record.
  • env var (environment variable) — a setting handed to a program when it runs. The SKIP_...=1 flags I use to bypass a pre-commit check are env vars.

8c. Database (where the 335K-source corpus lives)

  • SQL / query — the language for asking the database questions. SELECT ... WHERE ... = "give me the rows matching this."
  • schema — the database's structure: which tables exist and what columns each has. ("The thread has a t10_membership column" is a schema fact.)
  • migration — a recorded script that changes the schema (adds a column, etc.) in a controlled way, so the change is auditable.
  • index — a lookup shortcut that makes queries fast, like the index at the back of a book.
  • lock / busy_timeout — how the DB stops two writers from colliding: one waits while the other writes. busy_timeout = how long to wait before giving up. (This is why we have the daemon + the "one writer at a time" rule.)
  • daemon — a program that runs quietly in the background, all the time, doing one job whenever needed, never talking to you directly (pronounced "demon," from the Greek daimon = an unseen helper-spirit). Ours (db_writer_daemon) is the single orderly writer to the database — everything else queues its writes and the daemon applies them one at a time. Its invisibility is the feature and why it sat silently broken for 10 days (S102).
  • idempotent — an operation safe to repeat: running it twice gives the same result as once (so a retry can't double-apply).

8d. LLM / AI mechanics

  • SDK — a toolkit for calling a service from code. Ours calls Claude through your Max plan (so it's $0, never the paid API).
  • API — the "doorway" one program uses to talk to another service over the internet.
  • token — the unit LLMs read and write in — roughly ¾ of a word. Speed and cost scale with tokens, which is why "reduce the output" (fewer tokens) was the W9 lever that actually worked.
  • rate limit — the cap on how many calls you can make in a time-window (~1000 / 5 hours per Max plan). The real W9 bottleneck.
  • prompt — the text instruction sent to the model.
  • temperature — a randomness knob; 0 = as deterministic as possible (what we use for coding).
  • JSON — a structured text format for data — the shape the codings come back in ({"INSIDER": {...}}).
  • concurrency / serial / parallel — many things at once / strictly one-at-a-time / simultaneously. Today's surprise: on the bundled SDK, concurrency was slower than serial.
