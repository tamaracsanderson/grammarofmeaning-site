How we test this

The method is the contribution — so how we know it works is the argument, not housekeeping. Every result is checkable, organized by the question a reader asks: how do you know that? A generative engine is graded on aggregate behavior across runs; a deterministic step, on exact reproduction.

← back to the schema

Do independent coders agree, from the same logic? — reliability

Does locked-method × locked-model reproduce a coding?
Reliability / R-gradesR1 (high) / R2 (medium) / R3 (low) per (passage × dimension) cell — one passage can be R1 on one dimension, R3 on another. Generative steps graded on aggregate rates across runs, deterministic steps per-cell. Ex: Romans 8 — a silence was agreed present (α≈0.59) but not obligated (α≈0.03).
IRR family — 5IRR / 3IRR / 3PWhether independent LLM cohorts converge (they never see each other's answers). 5IRR = 5 web-UI cohorts → consensus buckets; 3IRR = 3 local models (cheap); 3P = 3-persona (insider / comparativist / practitioner). They compose (3P×5IRR = 15 reads). Ex: IRR167.

Does it hold beyond its home domain? — instrument vs. domain

Is a result a property of the instrument, or of the theology?
Out-of-domain replicationReligion is the proof-of-concept domain; secular corpora (a pop lyric, a translation set, a book vs. a passage) are confound-free test-beds — a clean test-bed is a floor, not a difficulty proxy. Ex: the Romans-8 present-vs-obligated split reproduced on a Beatles pop lyric (IRR167, 5/5), turning a suspected theology problem into a demonstrated instrument problem. Replication failures are findings, not embarrassments.

Is the data + spec sound, and actually built? — audit

Does the code match the claim, and the data match the schema?
Internal auditsData integrity + spec-vs-reality drift, run nightly / pre-commit / per-sprint: db-hygiene, state-audit, the spec-vs-implementation gap audit (specs with no code behind them), the nightly doc-drift auditor. Dashboards read the DB, not narration — they can't be talked past.
XT — programmatic testsAn empirical question run at $0, producing data (vs. an IRR, which produces judgments). Ex: XT1 cross-model persona (frame dominates, not model), XT4 move-segmentation reliability.

Does an independent re-derivation reproduce it? — clean-room

Can someone else get the same result, without our framing?
Clean-room reviewA finished artifact checked by an independent party who does not see the author's framing — catches contamination and bias. Worked example below.
Clean-room implementationWhether the method reproduces from its written spec alone, no tacit knowledge — an independent coder builds from the logic, outputs compared. The "logic, then code from it" test; it proves the method is not idiosyncratic.

Worked example — the data-dictionary audit, 2026-07-23

We ran a clean-room audit against our own data dictionary and led with the result rather than fixing it quietly, because a schema that shows what's wrong with it is worth more than one that looks finished. Three independent auditors, each given a lens that could fail differently, none seeing the others' work:

lenswhat it was allowed to seewhat only it could catch
vs. codethe SQL, the CHECK constraints, and every script that writes a columndefinitions that claim what the code doesn't do; columns defined but never written
cross-documentlexicon, decision journals, IRR reconciliations, design docsone term carrying two senses; a definition contradicted by a later lock
naive readeronly the definitions file — no code, no docs, no CLAUDE.mdanything only intelligible to someone who already knows the project

It found ~40 problems in 129 fields, including two factual errors introduced that same evening. The most consequential:

  • Nothing in the engine tables is validated, and the schema never said so. coding_status was undefined on all seven tables — while being the field that carries the answer: 100% of 2,600 rows are engine-proposed; zero are cold-confirmed.
  • Two primary-key definitions were false. Written hours earlier from a single template, they claimed "one row per source text" for tables holding 12.25 and 10.00 rows per source.
  • A field wrong for the third consecutive time. perturbed_axis had already been hand-corrected twice; the audit showed it still listed three axes the CHECK constraint makes unstorable.
  • A self-contradiction. One definition was annotated "the draft hallucinated this vocabulary — wrong"; the definition two lines below still cited that same vocabulary.
  • Vocabularies controlled in intent, enforced nowhere. The Sitz fields pass LLM output through unvalidated — conforming by luck, not by constraint.

Why three lenses and not one. The naive reader caught that two fields were indistinguishable; the code lens caught why — they hold byte-identical values in all 1,817 rows. Neither alone gives you the fix. And where the researcher's own walk, the naive reader, and the cross-document lens independently flagged the same two fields, that convergence is evidence the finding is real rather than one reader's confusion.

The generalizable rule this produced: a verification marker must live in the code that changed, not in the data it displays. A check proposed during this round would have passed against the old build, because the strings it searched for were present in both. A check that cannot fail is not a check.

Does the generated map match reality? — predictive

Do the outputs predict what actually happened?
Round-trip / predictive validityReverse-coding (code → back to text) + whether Engine 3's possible answer-space predicts Engine 4's attested reception. The generated field is the controlled variable; the attested record is the observed one. (Designed — Engines 3/4 not yet built.)

Could this check have failed? — discrimination

Guards against silent fallback — could this check have failed?
Discriminating-check ruleGuards against silent fallback — a step fails, the system substitutes something plausible, and the output still looks normal. The guard is one question asked of every check: could this have failed? A verification whose marker lives in the data being displayed rather than in the code that changed will pass against the old build and the new one alike. Fires whenever a fix is verified. Instances caught in a single session (2026-07-23): a constraint violation that made the pipeline fall back to a stale tree; a git stash that swallowed untracked work; a label truncated mid-word by a silent [:34]; a scheduled job installed but never loaded, so it had never once run; a 0-byte database file that answered "no such table" instead of "this file is empty." Each degraded quietly and kept producing plausible output.

Sibling to ghost-protection below: that one catches fabricated claims; this one catches unfalsifiable checks and quiet degradation.

Is a claimed past decision real? — ghost-protection

Guarding against fabricated provenance.
Citation-or-don't-claim + adversarial spot-checksAny "we decided X at session N" must cite a specific Decision / Proposal / journal entry, or it is flagged unverified. Adversarial verifiers are prompted to refute, not confirm. The dashboards above are the primary deterministic ghost-detector.
Why this is a contribution, not a checklist. Most comparative work publishes findings and withholds the apparatus. This project publishes the apparatus — so the checks are the open-methodology claim. Its honest counterpart is on the schema itself: the ⚠ undefined counts. Showing the gaps and showing the test suite are the same commitment.
